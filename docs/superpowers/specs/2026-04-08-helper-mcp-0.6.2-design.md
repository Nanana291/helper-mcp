# helper-mcp 0.6.2 Design

## Goal

Ship a large follow-up release that makes Luau analysis feed the brain automatically.

The release keeps the 0.6.1 Luau/brain/workspace surface intact, but adds a stronger bridge between Luau findings and durable brain notes so code intelligence becomes usable memory without a manual copy step.

## Scope

### In scope

- Expand Luau analysis so every command that produces findings emits normalized finding records.
- Auto-create or refresh one brain note per finding, including `high`, `review`, and useful `info` findings.
- Add brain commands for finding-focused history, graphing, querying, and cleanup.
- Keep canonical tool names and underscore aliases stable for the 0.6.1 surface.
- Keep the bridge deterministic so the same finding always maps to the same derived note identity.
- Preserve fail-closed behavior for malformed pattern overrides and unsupported analysis input.

### Out of scope

- Breaking canonical names or aliases that already exist in 0.6.1.
- Turning the bridge into a generic event bus or background daemon.
- Adding workspace automation as the main release theme.
- Making auto-created notes lossy or ephemeral.

## Architecture

### Shared Spine

The release uses one internal spine across Luau and brain:

- `finding normalization`
  - Every Luau analyzer returns a canonical finding shape with `severity`, `confidence`, `label`, `filePath`, `line`, and `evidence`.
- `brain bridge`
  - Normalized findings are written to brain as derived notes.
  - One finding maps to one note identity, so repeated scans refresh the same record instead of creating duplicates.
- `temporal proof`
  - Bridge activity is written into brain history so the source of a note can be reconstructed later.
- `regression gate`
  - Benign fixtures stay quiet.
  - Finding creation, note identity, status mapping, and dedupe behavior are explicitly tested.

### Pillar 1: Luau Intelligence

Luau commands should emit the same finding shape whether the result is structural, risky, or advisory.

Planned commands:

- `luau.findings`
  - Return normalized findings for a file or workspace slice.
  - This is the canonical shape used by the bridge and by command-level output.
- `luau.flow`
  - Summarize data flow between locals, functions, and remote calls.
- `luau.handlers`
  - Map remote events/functions to likely handlers and missing coverage.
- `luau.modulegraph`
  - Build a module dependency graph and identify disconnected or overly shared modules.
- `luau.complexity`
  - Score cyclomatic complexity for each function in a Luau file.
- `luau.taint`
  - Trace risky values from source to sink across a file or workspace slice.
- `luau.security_scan`
  - Detect webhook leaks, token exfiltration, remote loaders, and backdoor patterns.
- `luau.risk_score`
  - Produce a normalized score from risk, security, and performance signals.
- `luau.repair`
  - Suggest a conservative fix snippet for a named risk label.
- `luau.brain_sync`
  - Force a bridge pass for a file or workspace slice when a caller wants to backfill memory explicitly.

Design intent:

- Prefer normalized findings over command-specific ad hoc output.
- Keep `high`, `review`, and selected `info` findings bridgeable.
- Reuse the existing Luau parsing helpers and patterns infrastructure where possible.

### Pillar 2: Brain Memory

Brain commands should make finding-derived notes easy to inspect, trace, and clean up.

Planned commands:

- `brain.findings`
  - Query notes that were derived from Luau findings, with filters for severity, status, file path, and label.
- `brain.finding_history`
  - Show the temporal evolution of finding-derived notes and their status changes.
- `brain.finding_graph`
  - Show relationships between finding notes, regular notes, shared labels, and shared source files.
- `brain.finding_prune`
  - Find or consolidate duplicate finding notes using deterministic identity rules.

Design intent:

- Keep the bridge one-note-per-finding so memory is easy to scan and reason about.
- Preserve note history even when finding notes are updated or cleaned up.
- Make the brain useful as a durable audit trail for Luau results, not only as a manual note store.

## Bridge Rules

### Note creation

- `high` findings create active brain notes by default.
- `review` findings create candidate brain notes by default.
- `info` findings create candidate brain notes only when they come from structural commands that expose meaningful dependencies, graphs, or handler coverage.
- Each note includes:
  - source command
  - file path
  - line number
  - label
  - severity
  - confidence
  - evidence text
  - suggested fix when available

### Note identity

- The bridge uses a deterministic identity derived from the finding source, file path, line, label, and normalized evidence.
- Re-running the same analysis updates the existing note instead of creating a duplicate.
- If the source text changes materially, the bridge creates a new identity and leaves the old note in history.

### Status mapping

- `high` -> `active`
- `review` -> `candidate`
- `info` -> `candidate`
- A later manual brain action can still promote or archive the note.

## File Boundaries

- `src/luau.mjs`
  - Normalized finding extraction and bridge-friendly analysis helpers.
  - Luau analyzers emit finding records with severity and confidence.
- `src/brain.mjs`
  - Finding note creation, identity, history, query, graph, and cleanup helpers.
- `src/core.mjs`
  - MCP tool registry, aliases, handlers, versioning, and bridge orchestration.
- `src/patterns.mjs`
  - Pattern override loading and category normalization.
- `test/*.mjs`
  - Fixture-based behavior tests for Luau findings, brain bridge persistence, and compatibility coverage.
- `docs/releases/0.6.2/*`
  - Task lock, handoff, regression matrix, and delivery gate.

## Data Flow

### Luau flow

1. Read source text and workspace context.
2. Load project patterns with fail-closed overrides.
3. Analyze for flow, handlers, dependency shape, risk, and complexity.
4. Normalize findings into the common finding shape.
5. Return structured command output and pass the same findings to the brain bridge.

### Brain flow

1. Receive normalized findings from Luau commands.
2. Derive or reuse the deterministic note identity.
3. Create or refresh the matching brain note.
4. Append a history event for auditability.
5. Expose finding-specific queries and graphs for later review.

### Bridge flow

1. A Luau command returns findings.
2. The bridge filters findings by status mapping and bridgeability.
3. One note is written per finding.
4. Repeated runs refresh the same derived note when the identity is stable.
5. The caller still receives the normal analysis result, plus bridge metadata.

## Error Handling

- Malformed pattern overrides fall back to defaults.
- Empty or benign inputs return structured results with no bridge writes.
- Commands that cannot extract a finding identity should skip brain writes instead of guessing.
- Bridge writes should fail closed on missing file paths or unresolved workspaces.
- Duplicate detection should prefer deterministic identity over fuzzy merge guesses.
- Auto-write failures should surface clearly in the command result so analysis is not mistaken for persistence.

## Testing

### Luau tests

- Structural fixtures generate normalized findings with severity and confidence.
- Benign snippets remain quiet.
- Risk-heavy fixtures create bridgeable findings.
- `luau.brain_sync` writes a predictable note set.

### Brain tests

- Finding notes are created one per finding.
- Re-running the same analysis refreshes the existing note instead of duplicating it.
- `brain.findings` respects severity and label filters.
- `brain.finding_history` preserves status evolution.
- `brain.finding_graph` links finding notes to each other and to regular notes where appropriate.

### Regression proof

- Preserve `0.6.1` canonical behavior and aliases.
- Confirm the new bridge does not remove the existing surface.
- Keep `node --test` as the release gate.

## Release Criteria

The release is ready when:

- All Luau commands that emit findings also persist the intended brain notes.
- `high`, `review`, and bridgeable `info` findings are written with stable identities.
- Repeated scans refresh the same derived note instead of duplicating it.
- `brain.findings`, `brain.finding_history`, `brain.finding_graph`, and `brain.finding_prune` return structured output on representative fixtures.
- `node --test` passes.
- Release artifacts exist for `0.6.2`.
- The existing 0.6.1 tools still behave unchanged.
