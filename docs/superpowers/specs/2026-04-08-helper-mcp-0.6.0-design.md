# helper-mcp 0.6.0 Design

## Goal

Ship a stabilization release that makes the existing Luau and brain tooling easier to trust in real work.

The release has two primary goals:

1. Reduce false positives and noisy findings in Luau analysis.
2. Strengthen recovery, persistence, and change tracking so the workspace can be restored and audited with less manual work.

## Scope

### In scope

- Improve pattern handling so project-specific conventions can override defaults without breaking built-in detection.
- Add confidence metadata or equivalent ranking to Luau findings so the caller can distinguish strong signals from heuristics.
- Tighten recovery flows around baselines, changelogs, metrics, and hotfix snapshots.
- Improve brain note hygiene:
  - temporal history
  - deduplication
  - merge suggestions
- Expand regression coverage around benign snippets, edge cases, and restore paths.

### Out of scope

- Large new analysis families that require deep control-flow or taint tracking.
- Breaking changes to canonical tool names or existing compatibility aliases.
- UI work unrelated to persistence, auditability, or false-positive reduction.

## Proposed Architecture

### 1. Pattern Engine

Keep the current default Luau patterns, but make overrides first-class and explicit.

- Default patterns stay as the base behavior.
- `.helper-mcp/patterns.json` can override or replace categories per project.
- Pattern loading should fail closed:
  - unreadable or malformed override files fall back to defaults
  - invalid entries are ignored rather than poisoning the entire scan

This keeps analysis adaptable without requiring code edits for every project convention.

### 2. Confidence Layer

Each Luau finding should carry a severity and a confidence hint.

- Strong signals:
  - webhook leaks
  - remote calls without safety wrappers
  - obvious deprecated API usage
- Medium or advisory signals:
  - broad heuristics
  - structural guesses
  - dependency inference from `require()` patterns

The goal is not more output. The goal is better output ordering and better caller judgment.

### 3. Recovery Layer

Make the recovery path durable and inspectable.

- Hotfixes keep before/after snapshots.
- Baselines remain the source of truth for known-good state.
- Changelogs should be generated from actual diffs, not inferred summaries.
- Metrics should be append-only so trend direction is visible over time.

This layer should make it possible to answer:

- What changed?
- What was repaired?
- What got worse?
- What is safe to restore?

### 4. Brain Hygiene

Brain state should behave like an audit trail, not just a note bucket.

- `brain.history` should show changes in updated order.
- `brain.merge` should surface probable duplicates before they accumulate.
- Merges should be conservative and require explicit application.

The brain should help explain the work, not duplicate noisy versions of the same lesson.

### 5. Regression Gate

Add tests that specifically protect against false positives and silent recovery regressions.

- Benign Luau snippets should stay clean.
- Override files should change only the patterns they explicitly define.
- Recovery snapshots should be generated on write paths.
- Brain history and merge flows should preserve note identity and status changes.

## Data Flow

### Scan flow

1. Read the workspace.
2. Load default patterns.
3. Overlay project-specific patterns if present.
4. Analyze files.
5. Attach confidence and severity metadata.
6. Emit the current scan result.

### Recovery flow

1. Compare current content against the known-good baseline or old version.
2. Generate a changelog or migration note.
3. Persist the note to the brain.
4. If a repair is applied, store before/after snapshots.
5. Update metrics if the tool path is metric-aware.

### Brain flow

1. Append or import notes.
2. Record temporal events when notes are updated, promoted, merged, or deleted.
3. Use search scoring to identify likely duplicates.
4. Let merges remain explicit so dedupe stays safe.

## Error Handling

- Invalid pattern override files should not block analysis.
- Missing file paths should fail with a clear error rather than a partial result.
- Merge operations should refuse to apply if there are not enough notes selected.
- Repair suggestions should prefer safe snippets over aggressive rewrites.
- Recovery tools should keep writing snapshots even if the analysis is partial, as long as the target file exists and is readable.

## Testing

### Required coverage

- False-positive tests on benign Luau snippets.
- Pattern override tests for `.helper-mcp/patterns.json`.
- History tests for note update/promotion ordering.
- Merge tests for duplicate note detection and explicit application.
- Recovery tests for baseline, hotfix, changelog, and metrics paths.

### Regression proof

- Preserve canonical tool names and compatibility aliases.
- Assert the tool count and resource count exposed by the MCP server.
- Confirm the existing 0.5.5 tools still work after the stabilization changes.

## Release Criteria

The release is ready when:

- `node --test` passes.
- False-positive fixtures remain clean.
- Recovery snapshots and changelogs are written for the intended paths.
- Brain history and merge flows produce auditable results.
- No canonical tool names are broken.
